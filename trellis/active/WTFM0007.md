# WTFM0007 - Help manifest and anchor drift check

Anchor drift is the URL-help system's silent failure mode: a heading id that
stops matching an editor `field-name` breaks that field's help with no error
anywhere (called out in Taproot TR00385). This task makes the contract
checkable by machines on both sides.

## Scope

- **Help manifest output.** At build time, emit a `help-manifest.json`
  describing every surface (WTFM0005): slug, reference-page URL,
  help-document URL (WTFM0006), and the ordered list of anchor ids the help
  document exposes.
- **Check command.** A consumer-runnable check (extend the existing
  `validate-manifest` export or add e.g. `@taprootio/wtfm/check-help-anchors`)
  that takes an expected-anchors file — e.g. Taproot exports its editor
  `field-name`s per surface — and exits non-zero listing any expected anchor
  missing from the built output, and optionally any orphaned extras.
- Document the expected-anchors file format and the intended CI wiring (the
  consumer runs the check after its docs build; Taproot would run it in the
  TR00384 site's workflow).
- Tests: manifest shape from a fixture build; check passes/fails correctly on
  matching, missing, and orphaned anchors.

## Notes

- The manifest doubles as a link index for anything that wants to construct
  help URLs programmatically (e.g. the editor wiring in Taproot TR00386)
  instead of hardcoding paths.
- Keep the check read-only and fast — it gates consumer CI, not wtfm's.

## Risks

- Expected-anchor lists living in the consumer can themselves go stale
  against the *editor* — the check closes the docs-side gap; pairing
  field-name changes with anchor updates remains a consumer convention
  (Taproot TR00385 keeps anchors under test beside the field names).
