# WTFM0006 - Help-document output mode conforming to ESP0119 strip-to-bare

wtfm's reference output is demo-first: descriptions render into interactive
`<wtfm-code-block>` elements carrying base64-encoded source and embedded JSON
scripts, wrapped in styled `doc-header`/`doc-section` divs. Espalier's
URL-help renderer (**ESP0119**) strips fetched help HTML to bare semantic
elements — that output would be gutted. This task adds the **help-document
mode**: prose-first output shaped to be fetched and injected into an
`esp-flyout`.

## Scope

- **A second render path** (e.g. a `renderHelpDocs(surfaceSlug)` shortcode or
  a `mode: "help"` renderer set) producing one help document per surface
  (WTFM0005), coexisting with the reference output in the same Eleventy
  build — the docs site serves both.
- **Markup contract (the ESP0119 keep-list):** headings, prose, lists,
  tables, code, blockquotes, images, and links only, plus anchor `id`s and
  `href`/`src`/`alt`. Concretely: no `<wtfm-code-block>` (render ```html
  examples as plain fenced code or omit them per-section), no wrapper-div or
  class-dependent structure, nothing that needs scripts or styling to make
  sense.
- **Anchors per field:** each member component / documented field renders as
  a section whose heading id follows WTFM0004's convention (or `@helpAnchor`
  override) so `help-src#<field-name>` resolves. One document serves a whole
  surface, so per-field help shows its section with surrounding context.
- **Link/image URL shape:** emit prefix-absolute `href`/`src` (coordinate
  with WTFM0008) — ESP0119 injects fetched markup into the *editor's* page,
  where document-relative URLs would resolve against the wrong base.
- Tests: a fixture surface rendered in help mode, asserted against the
  keep-list (no disallowed elements/attributes) and expected anchor ids.

## Notes

- Validate real output against the ESP0119 renderer **as released**, not the
  provisional API in its backlog item (mirrors Taproot TR00386's rule).
  Espalier's ESP0119 is active in the taproot-controls Trellis backlog.
- Help documents are trusted, app-owned content — ESP0119 normalizes but does
  not sanitize; wtfm's job is shape, not security.
- Consumed by Taproot TR00385 (authoring help content) and TR00386 (editor
  adoption).

## Risks

- Keep-list mismatch: too rich and the flyout strips meaning away, too lean
  and tables/code lose fidelity — test against the actual normalization
  early, not after authoring a full content set.
- Dual-output complexity: reference and help modes share renderers where
  possible; avoid forking section logic into two divergent code paths.
