# WTFM0009 - Retarget the client runtime at @taprootio/espalier

The Espalier design system publishes publicly as `@taprootio/espalier` (2.8.0),
but wtfm's client runtime still imports `EspalierElementBase` from the legacy
`@taprootio/taproot-controls` name and declares it as an optional peer
dependency. That legacy package exists only on GitHub Packages, and bun
materializes optional peers — so Espalier's own lockfile keeps resolving a
GitHub Packages copy of itself through wtfm, which is the one thing preventing
that repo from deleting its `@taprootio` scope routing and finishing the
public-registry migration WTFM0001 deferred.

## Scope

- `src/client/wtfm-code-block.ts`: import `EspalierElementBase` from
  `@taprootio/espalier` (exported from that package's root).
- Rename `src/client/taproot-controls.d.ts` to `src/client/espalier.d.ts` and
  redeclare the ambient module as `@taprootio/espalier`.
- `package.json`: rename the optional peer to
  `"@taprootio/espalier": ">=2.0.0"` (update `peerDependenciesMeta` to match)
  and bump the version to `0.15.0` — the rename is breaking for any client-
  runtime consumer still on the legacy name, and merging triggers the release.
- Sync the lockfile and add a CHANGELOG entry.

## Notes

- Downstream: after 0.15.0 ships, the Espalier repo can bump wtfm, delete its
  `.npmrc` scope routing, refresh `bun.lock`, drop GitHub Packages auth from
  CI, and the old GitHub Packages wtfm versions can be deprecated (the
  WTFM0001 loose end). That migration lives in the Espalier repo.
- Server-side output and the Eleventy plugin are unaffected — this touches
  only the client runtime's peer relationship.

## Acceptance criteria

- No reference to `@taprootio/taproot-controls` remains in `src/` or
  `package.json` (historical task files keep theirs).
- `npm ci && npm test` and `npx tsc --noEmit` pass with the renamed ambient
  stub.
- Merging publishes `0.15.0` to public npm via the existing OIDC workflow.
